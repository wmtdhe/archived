# 宏任务和微任务

# 宏任务引入

在JS中，大部分任务都是在主线程上执行的，常见任务有：

  1. 渲染事件
  2. 用户交互事件
  3. JS脚本执行
  4. 网络请求，文件读写完成事件

为了让这些事情有条不紊地进行，JS引擎需要对其执行顺序做一定安排，V8其实采用的是一种队列的方式来存储这些任务 —— 先来先执行。

宏任务包含了两种，除了普通的队列还有一个延迟队列，用来处理setTimeout/setInterval这样的定时器回调。

# 微任务引入

对于每个宏任务而言，其内部都有一个微任务队列。

微任务引入初衷：解决异步回调。

V8采用了将异步回调放在当前宏任务末尾的方式。在每一个宏任务中定义一个微任务队列，当该宏任务执行完成，会检查其中的微任务队列，如果为空则执行下一个宏任务，如果不为空，则依次执行微任务。

常见微任务有：MutationObserver, Promise.then(), Promise为基础开发的其他技术（如 fetch API），还有V8垃圾回收机制。

# EventLoop —— 浏览器篇

首先举个例子：

console.log('start');
setTimeout(()=>{
    console.log('timeout')
})
Promise.resolve().then(()=>{
    console.log('resolve')
})
console.log('end');
// start end resolve timeout

分析：

  1. 刚开始整个脚本作为一个宏任务来执行，对于同步代码直接压入执行栈进行执行，因此先打印出start和end
  2. setTimeout作为一个宏任务进入宏任务队列
  3. Promise.then()作为一个微任务进入微任务队列。
  4. 当本次宏任务执行完后，检查微任务队列，发现Promise.then,执行
  5. 进入下一个宏任务，setTimeout，执行。 因此打印顺序为：
// start end resolve timeout

更为完整的一个EventLoop过程：

  1. 一开始整段脚本作为第一个宏任务执行
  2. 执行过程中，同步代码直接执行，宏任务进入宏任务队列，微任务进入微任务队列
  3. 当前宏任务执行完成时，检查微任务队列，如果有，则依次执行，知道微任务队列为空
  4. 执行浏览器UI线程的渲染工作
  5. 检查是否有Web Worker任务，有则执行
  6. 执行队首昕的宏任务，回到步骤2，...知道宏任务和微任务都为空

练习：

Promise.resolve().then(()=>{
    console.log('Promise 1');
    setTimeout(()=>{
        console.log('setTimeout 2')
    })
});
setTimeout(()=>{
    console.log('setTimeout 1');
    Promise.resolve().then(()=>{
        console.log('Promise 2')
    })
})
console.log('start')
// start
// Promise 1
// setTimeout 1
// Promise 2
// setTimeout 2
