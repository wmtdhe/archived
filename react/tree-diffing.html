# 动机

使用React的时候，render()函数创建了一个React元素树。当在下一状态或者属性更新时，render()函数会返回一个不同的React元素树。接着React需要了解怎么有效快速的更新UI来匹配之前最新的树。

目前已有一些通用的方法来解决从一个树转换到另一个树所需要的最小的操作数量的算法问题。但这个state of the art algorithms的复杂度有O(n^3)，n为一个树中的元素数量。

如果我们在React中使用这个算法，展示1000个元素则需要十亿次比较。这个消耗过于庞大。取而代之，React实现了一种基于两个假设，复杂度大概为O(n)的算法：

  1. 两个不同类型的元素会生成不同的树
  2. 开发者可以通过key属性来提示那些子元素在不同的render中是保持稳定的

在实际运用中，这两个假设对于大多数的运用都是成立的。

# Diffing算法

当比较两个树的时候，React首先对比两个树的根元素(root)，根据不同类型的根元素，diffing算法有着不一样的行为。

# 不同类型的元素 Elements Of Different Types

只要是根元素有着不一样的类型，React就会摧毁旧树并建立新树。如从<a>到<img>，从<Article>到<Comment>，或是从<Button>到<div> - 这些都会引发一个完全的重建。

当摧毁一个旧树时，旧的DOM结点会被摧毁。组件实例会接收到componentWillUnmount()。当构建一个新树的时候，新的DOM结点会被插入到DOM中。组件实例会接收到componentDidMount()。任何关于旧树的状态都被丢弃了。

任何在根元素下的组件和其状态也跟随着根元素一同被摧毁了。举个例子来说，当对比以下两个组件时：

<div>
    <Counter />
</div>

<span>
    <Counter />
</span>

这个对比会将旧的Counter组件摧毁并重新挂在一个新的。

# 相同类型的DOM元素 DOM Elements Of The Same Type

当比较同类型的两个React DOM元素时，React会查看两者的属性 attribute，保留相同的DOM结点，只更新变更了的属性。比如：

<div className="before" title="stuff" />

<div className="after" title="stuff" />

对比这两个元素，React知道只用更改在DOM节点上的className就好。

当更新style时，React也会明白只用更新变更当属性就好。例如：

<div style={{color: 'red', fontWeight: 'bold'}} />

<div style={{color: 'green', fontWeight: 'bold'}} />

当在这两者之间转换时，React只会更改color当样式而不会更改fontWeight。

在处理完这个DOM结点后，React会接着递归子元素。

# 相同类型的组件元素 Component Elements Of The Same Type

当一个组件更新时，实例会保持不变，所以状态也会在各个渲染间被保存下来。React会更新组件实例的props以此来匹配新的元素，并且在这个组件上调用componentWillReceiveProps()和componentWillUpdate()。

接下来，render()函数会被调用，diff算法会在之前和现在的结果上递归。

# 子元素上的递归 Recursing On Children

默认情况下，当递归一个DOM结点的子元素时，React只会同时遍历两个列表的子元素，并且在发现不同的时候产生一个改变 mutation。

比如说，当在子元素末尾添加一个新元素时，新旧树之间的转换是很方便的简单的works well。

<ul>
  <li>first</li>
  <li>second</li>
</ul>

<ul>
  <li>first</li>
  <li>second</li>
  <li>third</li>
</ul>

React会匹配两个<li>first</li>树和两个<li>second</li>树，然后再插入<li>third</li>树。

但是，如果你从子元素开端插入一个新元素的话，则会产生很差的性能。举例说：

<ul>
  <li>Duke</li>
  <li>Villanova</li>
</ul>

<ul>
  <li>Connecticut</li>
  <li>Duke</li>
  <li>Villanova</li>
</ul>

React会改变 mutate每一个子元素而不是发现它可以保持<li>Duke</li>和<li>Villanova</li>子树不变。这会导致效率降低。

# Keys

为了解决上述出现的问题，React支持一个key属性。当子元素有keys时， React会使用这些key来匹配原始树中的子元素和接下来的树中的子元素。举例来讲，在上述性能差的例子中添加key属性可以是的树的转换变得效率起来：

<ul>
  <li key="2015">Duke</li>
  <li key="2016">Villanova</li>
</ul>

<ul>
  <li key="2014">Connecticut</li>
  <li key="2015">Duke</li>
  <li key="2016">Villanova</li>
</ul>

现在React会明白key值为2014的元素是新的元素，key值为2015和2016的元素只是被移动了位置。

在实际运用中，找到一个合适的key值并不困难。可能你需要展示的元素已经有了一个自己独有的ID，所key值可以是简简单单从你的数据中而来：

<li key={item.id}>{item.name}</li>

如果你的数据没有自己独有的ID的话，你也可以自己添加一个ID属性到你的模型里或是哈希部分内容来生成自己的key。key只用在兄弟元素之间保持独有就行。

最后，你还可以将元素的下标当作key。如果这个数组里的元素顺序不会改变(never reorder)的话性能会很好，反之性能则会变差。

当以下标作为key时，重新排序(Reorders)有时也会对组件的状态产生问题。组件实例的更新和复用是基于它们的key值的。如果key是下标的话，移动某个元素则会改变这个下标。造成结果就是，组件的状态对于某些东西来说 - 如非受控输入元素 uncontrolled inputs，会产生一些混淆，以一种意想不到的方式进行更新。

# 让步 Tradeoffs

对于每一次操作而言，React都可以重新渲染整个app，最后的结果也会是一样的。这里的重新渲染(rerender)指的是对所有组件调用render函数，而不是说React会摧毁并重建它们。只有符合之前所说的两个原则的改动才会进行摧毁和重建。

因为React依赖于估量方法，如果估量方法的假设没有被满足，性能则会受困：

  1. Diff算法不会尝试匹配不同组件类型的两个子树，如果你打算交替使用两种组件类型生成相似的结果的话，最好还是使用相同类型组件为好。
  2. key值应该是稳定，可预测，独特的。不稳定的key值（如通过Math.random()产生的）会导致许多组件实例和DOM结点被不必要的重复创建，因此导致性能下降和子组件的状态丢失。
